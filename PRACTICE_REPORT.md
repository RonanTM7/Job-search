## Задание 1. Компаративный анализ коммерческих приложений-агрегаторов и стратегия рыночного позиционирования

### 1.1. Анализ текущей ситуации на рынке интернет-рекрутинга в IT
Современный рынок труда в сфере информационных технологий предъявляет особые требования к инструментам поиска работы. В условиях высокой конкуренции за таланты и огромного количества предложений, универсальные площадки начинают проигрывать специализированным нишевым решениям. Основная проблема крупных агрегаторов заключается в "информационном шуме" — соискатель вынужден просеивать сотни нерелевантных объявлений.

**Развернутый анализ ключевых игроков:**

1.  **HeadHunter (hh.ru):** Безусловный лидер российского рынка по охвату.
    *   *Преимущества:* Огромная база работодателей, наличие системы отзывов, развитые алгоритмы ранжирования.
    *   *Недостатки:* Процесс заполнения резюме крайне трудоемок и содержит множество полей, не актуальных для IT (например, детальное описание хобби или нерелевантного опыта 10-летней давности). Мобильное приложение ориентировано на массового пользователя, что делает поиск по специфическим тегам (например, "Kubernetes" или "Golang") менее точным из-за особенностей индексации.
2.  **SuperJob:** Один из старейших сервисов.
    *   *Преимущества:* Высокая степень доверия со стороны крупных государственных и окологосударственных структур.
    *   *Недостатки:* Интерфейс и логика взаимодействия (UX) значительно отстают от современных стандартов IT-индустрии. Отсутствие глубокой интеграции с техническими портфолио делает платформу малопривлекательной для разработчиков уровней Middle и Senior.
3.  **Авито Работа:** Агрегатор, ориентированный на быстрый наем.
    *   *Преимущества:* Простота публикации и высокая скорость отклика.
    *   *Недостатки:* Платформа воспринимается как площадка для рабочих специальностей. IT-вакансии здесь часто представлены позициями с низким уровнем оплаты или сомнительными предложениями, что создает негативный фон для профессионального сообщества.
4.  **Хабр Карьера:** Ближайший по духу конкурент.
    *   *Преимущества:* Глубокая интеграция с крупнейшим IT-сообществом Хабр, наличие оценок компаний сотрудниками.
    *   *Недостатки:* Приложение зачастую воспринимается как дополнение к веб-версии, а не как самостоятельный инструмент для оперативного поиска.

### 1.3. Обоснование рыночной ниши и концепция уникальности
На основе проведенного анализа выявлена незанятая ниша **мобильных микро-агрегаторов**, которые позволяют пользователю не просто искать работу, а делать это в максимально лаконичном, "чистом" интерфейсе без лишней рекламы и нерелевантного контента.

**Критерии уникальности программного продукта:**
*   **Концепция "Zero-Waste UI":** Интерфейс спроектирован так, чтобы от момента запуска приложения до отклика на релевантную вакансию проходило не более 3-х кликов.
*   **Интегрированная роль модератора:** В отличие от конкурентов, где администрирование происходит через громоздкие веб-интерфейсы, наше приложение позволяет владельцу продукта управлять вакансиями, пользователями и чатами прямо со смартфона.
*   **Сфокусированный парсинг:** Мы не просто собираем всё подряд, а фильтруем предложения по ключевым IT-маркерам еще на этапе загрузки в облако.

### 1.4. Прогноз развития и SWOT-анализ
*   **Сильные стороны:** Низкий порог входа, высокая скорость работы, использование современных облачных технологий (Firebase).
*   **Возможности:** Внедрение системы рекомендаций на основе машинного обучения для сопоставления стека технологий в резюме и вакансии.
*   **Угрозы:** Потенциальное ограничение доступа к данным со стороны крупных площадок-источников.

## Задание 2. Анализ требований к инфраструктуре поддержки приложения

### 2.1. Методология выбора инфраструктурных решений
Для реализации современного мобильного приложения выбрана облачная архитектура. Это обусловлено необходимостью обеспечения высокой доступности (High Availability) и возможностью быстрого масштабирования при росте базы пользователей без необходимости физической настройки серверов.

### 2.2. Требования к среде разработки и целевой платформе
*   **Аппаратное обеспечение разработчика:** Для комфортной компиляции и работы с эмуляторами Android требуется станция с процессором не ниже Intel i7/Ryzen 7, 16 ГБ оперативной памяти и быстрым NVMe накопителем.
*   **Программный стек:**
    *   ОС: macOS, Linux или Windows 11.
    *   IDE: Android Studio (последняя стабильная версия).
    *   Языки: Java 17 для обеспечения совместимости с большинством библиотек Firebase.
*   **Целевая платформа:** Android API Level 24+ (Nougat и выше). Это позволяет охватить более 94% активных устройств на рынке, сохраняя доступ к современным API для работы с сетью и графикой.

### 2.3. Хостинговая инфраструктура на базе Google Firebase
Приложение является Serverless-решением, что радикально снижает затраты на поддержку.
1.  **Cloud Firestore (Система хранения):** NoSQL база данных с поддержкой реального времени.
    *   *Структура:* Документоориентированная модель. Данные кэшируются локально, что позволяет приложению работать в условиях нестабильного интернета.
2.  **Firebase Authentication:** Обеспечивает безопасное хранение паролей и управление сессиями. Поддерживает протоколы шифрования промышленного стандарта.
3.  **Cloud Storage:** Инфраструктура для хранения медиа-контента. Все файлы защищены правилами доступа, привязанными к уникальному идентификатору пользователя (UID).

### 2.4. Подсистема автоматизированного сбора вакансий (Парсер)
Для наполнения приложения данными спроектирован модуль, работающий независимо от основного мобильного клиента.
*   **Серверная среда:** Виртуальный сервер (VPS) с 2 ядрами CPU и 4 ГБ RAM.
*   **Стек парсера:** Python 3.x с использованием асинхронных библиотек (aiohttp/Scrapy). Это позволяет обрабатывать сотни страниц внешних сайтов в минуту.
*   **Сетевая безопасность:** Использование распределенной сети прокси-серверов позволяет имитировать поведение реальных пользователей, предотвращая блокировку IP-адреса инфраструктуры.

### 2.5. Анализ нагрузки и расчет масштабируемости (1000 DAU)
Расчеты производились исходя из среднего поведения пользователя (просмотр 15 вакансий, 2 отклика, 5 сообщений в чате):
*   **Чтения БД:** ~15,000 - 20,000 операций в сутки.
*   **Записи БД:** ~2,000 - 3,000 операций в сутки.
*   **Трафик:** Суммарно около 5-7 ГБ в месяц (с учетом загрузки изображений).
*   **Масштабирование:** При достижении лимитов бесплатного уровня, переход на план "Blaze" (Pay-as-you-go) позволяет бесшовно обрабатывать до 100,000+ пользователей.

### 2.6. Информационная безопасность
*   **Encryption:** Весь трафик между приложением и Firebase шифруется по протоколу TLS 1.3.
*   **Security Rules:** Правила на стороне Firebase жестко ограничивают права доступа: `allow write: if request.auth.uid == userId`. Это гарантирует, что один пользователь не сможет изменить данные другого.

## Задание 3. Проектирование логической архитектуры программного продукта

### 3.1. Архитектурный паттерн и декомпозиция системы
В основе приложения лежит принцип разделения ответственности. Логическая архитектура спроектирована таким образом, чтобы изменения в интерфейсе не затрагивали логику работы с данными.

**Основные программные уровни:**
1.  **View Layer (Слой представления):** Состоит из Activities и Fragments. Использует XML-разметку для формирования интерфейса. Включает компоненты Material Design для обеспечения современного внешнего вида.
2.  **Business Logic Layer (Слой логики):** Содержит контроллеры и вспомогательные классы (Utils). Здесь происходит валидация вводимых данных (например, проверка формата email или номера телефона), а также обработка бизнес-правил (например, проверка прав администратора).
3.  **Data Access Layer (Слой данных):** Инкапсулирует работу с Firebase. Использует паттерн Repository для предоставления данных остальным частям приложения.

### 3.4. Описание интерфейсов и протоколов взаимодействия
*   **gRPC / Protobuf:** Firebase SDK использует высокопроизводительный протокол gRPC для обмена данными с облаком Firestore. Это обеспечивает минимальный объем передаваемого трафика и мгновенные уведомления об изменениях (Push-модель).
*   **REST API:** Используется модулем парсинга для взаимодействия с внешними сайтами-агрегаторами и API прокси-серверов.
*   **Local Persistence Interface:** Внутренний интерфейс приложения для работы с кэшем. Позволяет сохранять состояние интерфейса и данные вакансий в SQLite-базе (через Firebase Offline), обеспечивая работу приложения без сети.
