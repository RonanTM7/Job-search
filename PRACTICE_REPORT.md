## Задание 1. Компаративный анализ коммерческих приложений-агрегаторов

### 1.1. Анализ конкурентов
В рамках анализа были рассмотрены ключевые игроки рынка: **HeadHunter (hh.ru)**, **SuperJob**, **Работа.ру** и **Авито Работа**.

*   **HeadHunter:** Основной конкурент с огромной базой. Однако интерфейс перегружен для мобильного пользователя, а фокус на всех специальностях сразу мешает быстрому поиску именно в IT.
*   **SuperJob:** Обладает хорошей базой, но мобильное приложение менее интуитивно понятно и редко обновляется в плане UI-трендов для разработчиков.
*   **Авито Работа:** Высокий уровень шума из-за обилия вакансий для линейного персонала (курьеры, водители).

### 1.2. Матрица функций

| Функция | hh.ru | SuperJob | Работа.ру | Авито Работа | **Наш проект (IT-Job)** |

### 1.3. Ниша и уникальность
**Ниша:** Специализированный, "легкий" мобильный агрегатор вакансий исключительно для IT-сферы.
**Критерии уникальности:**
1. **Zero Noise:** Отсутствие вакансий не связанных с IT.
2. **Mobile Admin:** Возможность управления базой вакансий и пользователями прямо из мобильного клиента (уникально для данного сегмента).
3. **Speed to Hire:** Прямой чат с администратором/модератором для быстрой помощи соискателям.

## Задание 2. Анализ требований к инфраструктуре

### 2.1. Среда разработки
* **IDE:** Android Studio (версия Hedgehog или новее).
* **Язык:** Java 17 / Kotlin.
* **Платформа:** Android API 24+ (7.0 и выше).

### 2.2. Хостинг и Backend (Firebase)
* **Authentication:** Для регистрации и авторизации.
* **Cloud Firestore:** В качестве основной NoSQL базы данных.
* **Firebase Storage:** Для хранения файлов резюме и изображений профиля.

### 2.3. Сервер парсинга
* **Технологии:** Python (библиотеки `requests`, `BeautifulSoup`, `Selenium`).
* **Требования к серверу:** 2 vCPU, 4 ГБ RAM (минимум для работы Headless-браузеров), 40 ГБ SSD.
* **Прокси:** Необходимы для обхода лимитов внешних агрегаторов.

### 2.4. Нагрузка (1000 DAU)
При 1000 активных пользователях в день:
* **Запросы к БД:** Около 20,000 операций в сутки.
* **Трафик:** Примерно 1.5 ГБ в месяц.
Это полностью покрывается мощностями Firebase и не требует выделенных физических серверов на начальном этапе.

## Задание 3. Логическая архитектура

### 3.1. Диаграмма вариантов использования (Use Case)
Для просмотра в Draw.io: **Advanced -> Mermaid**.

```mermaid
usecaseDiagram
    actor "Пользователь" as User
    actor "Администратор" as Admin

    User --> (Поиск вакансий)
    User --> (Создание профиля/резюме)
    User --> (Отклик на вакансию)
    User --> (Чат с поддержкой)

    Admin --> (Управление пользователями)
    Admin --> (Модерация вакансий)
    Admin --> (Просмотр статистики)
    Admin --> (Ответ в чате)
```
